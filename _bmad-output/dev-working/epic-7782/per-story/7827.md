# E6-S3 — pm-service core

One-pass close-out, zero strikes. Seven source files + one test file under `src/lib/pm-service/`.

## Contracts locked for downstream

- **AssignInput** shape (S5 must pass verbatim): `submissionId, referralCode, customerId, userId, propertyId, pillarHint, seller:{fullName,email,phone?,address,city,state,zip,beds?,baths?,sqft?,yearBuilt?,timeline?,sellerPaths[],tcpaVersion?,tcpaAcceptedAt?,termsVersion?,termsAcceptedAt?}, offers:[{path,lowCents,highCents,rawPayload}]`.
- **AssignResult** discriminated union:
  - Success: `{ ok:true, pmUserId, pmFirstName, pmPhotoUrl, profileCreated, emailsEnqueued:{seller,team} }`
  - Failure: `{ ok:false, reason:'no_active_pms'|'timeout'|'db_error'|'profile_failed'|'submission_failed'|'unexpected', sentryEventId? }`
- **Orchestrator never throws** — always returns `AssignResult`.
- **Sentry event names:** `pm_assignment_failed` (critical for `no_active_pms`, else error) + `pm_email_failed` (error). Locked as S8 alert contract.
- **PII scrub:** `sanitizeSentryExtras` allow-list in `observability.ts`. Extending it = explicit, diffable change.
- **PmPreview shape:** `{ firstName, photoUrl }`. No lastName, no contact info.
- **S4 hand-off:** `email-stub.ts` is deleted by S4; barrel re-points to `src/lib/email/`.

## Decisions worth remembering

- Added two reasons beyond the spec (`profile_failed`, `submission_failed`) to distinguish pre-RPC failures. Both map to `pm_assignment_failed` so S8's alert rule catches them without extending its filter.
- Supabase client stays untyped — the `Database` generic in supabase-js v2.104 fights additive schema evolution and imposes Views/Enums/CompositeTypes shape requirements that aren't worth satisfying for this MVP. Row types remain exported from `schema.ts`.
- Team-member email recipient uses a stable placeholder (`team+${id}@sellyourhousefree.internal`) until S4 wires real routing via `EMAIL_REPLY_TO`.
