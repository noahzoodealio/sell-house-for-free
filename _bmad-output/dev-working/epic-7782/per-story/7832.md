# E6-S5 — actions.ts call-site wiring

Wires the S3 orchestrator into the real E5 submit flow. The PM-handoff stub that was a logAudit placeholder is replaced with a live `assignPmAndNotify` call.

## Flow

After Offervana success:
1. `fetchAndLogOffers` returns raw offers (now returning `unknown[] | null` instead of void).
2. `runPmHandoff(draft, payload, offers)` builds `AssignInput` and calls `assignPmAndNotify`.
3. Orchestrator upserts `submissions` + `submission_offers` (idempotent on submission_id + (submission_id, path)), creates auth.users + profiles, calls `assign_next_pm` RPC, sends emails via `Promise.allSettled`.

## Deviations from filed story body

- The filed body said "exactly one file edited ... one import + one await ... + two console.log lines". Reality was larger: I had to refactor `fetchAndLogOffers` to return the offers (it was previously void) so the call site has offers in hand when calling assignPmAndNotify, and I added a `buildAssignInput` helper + a `offersFromPayload` mapper. Net diff +127 / -14 on a single file.
- Redirect was already `/portal/setup?sid=...&ref=...` (E3/E9 already landed it). No URL changes needed.
- `userId: null` on AssignInput — Offervana's OkPayload type only exposes `customerId`, `referralCode`, `propertyId`. No user ID field.

- `[e6.assign].ok` — pmUserId, profileCreated, sellerEmailEnqueued, teamEmailEnqueued.
